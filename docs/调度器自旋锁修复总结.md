# 调度器自旋锁问题修复总结

## 问题描述

原有的调度器实现使用 `should_stop_` 自旋标志配合分段轮询来检查停止信号，存在以下严重问题：

### 1. **竞态条件**
- `SchedulerManager::stop_scheduler()` 设置 `should_stop_=true`
- 但调度器线程可能刚检查完标志，继续执行下一轮循环
- 导致停止信号被错过

### 2. **双重标志不一致**
- `running_` 和 `should_stop_` 两个标志没有同步更新
- `stop()` 方法只检查 `running_`，不检查 `should_stop_`
- 状态管理混乱

### 3. **低效的轮询等待**
```cpp
// 原有实现 - 每 5ms 检查一次
while (remaining_ms > 0 && !should_stop_) {
    int sleep_segment = std::min(5, remaining_ms);
    std::this_thread::sleep_for(std::chrono::milliseconds(sleep_segment));
    remaining_ms -= sleep_segment;
}
```
- CPU 浪费在轮询检查上
- 响应延迟最高可达 5ms

### 4. **内存可见性问题**
- `should_stop_` 虽然是 `atomic<bool>`，但不是内存屏障
- 睡眠期间的修改可能不会立即可见

---

## 解决方案：采用condition_variable

选择 **condition_variable** 方案的原因：
1. ✅ **高效等待**：线程睡眠而非轮询，零 CPU 浪费
2. ✅ **立即响应**：可被 `notify_one()` 瞬间唤醒（微秒级）
3. ✅ **标准实践**：C++ 并发编程最佳实践
4. ✅ **易于扩展**：支持超时、条件等待等高级功能
5. ✅ **符合策略模式**：与现有架构完美融合

#### 2. 实现文件（4 个调度器）

- `src/vm/Scheduler/Hybrid_Scheduler.cpp`
- `src/vm/Scheduler/QuotaScheduler.cpp`

**stop() 方法改进**：
```cpp
void stop() {
    if (!running_.exchange(false)) {
        return;
    }
    
    // 关键：唤醒等待中的调度线程
    cv_.notify_one();  // ← 立即唤醒
    
    if (scheduler_thread_.joinable()) {
        scheduler_thread_.join();
    }
}
```

**scheduler_loop() 重构**：
```cpp
// ❌ 旧实现 - 自旋轮询
while (running_ && !should_stop_) {
    // 执行调度...
    
    // 分段睡眠 + 轮询
    while (remaining_ms > 0 && !should_stop_) {
        std::this_thread::sleep_for(std::chrono::milliseconds(5));
    }
}

// ✅ 新实现 - condition_variable
while (running_) {
    // 执行调度...
    
    if (remaining_ms > 0) {
        std::unique_lock<std::mutex> lock(sched_mtx_);
        // 高效等待，可被 notify_one() 立即唤醒
        cv_.wait_for(lock, std::chrono::milliseconds(remaining_ms));
    }
}
```

## 核心优势总结

### 1. **线程安全**
- `condition_variable` 提供完整的内存屏障保证
- 配合 `unique_lock` 自动管理锁状态
- 消除了原子操作的内存序问题

### 2. **零性能开销**
- 等待时线程完全休眠，不占用 CPU
- 唤醒是内核级别的信号传递，微秒级响应
- 相比每 5ms 轮询一次，效率提升巨大

### 3. **设计优雅**
- 统一的停止机制：只需调用 `stop()`
- 移除了冗余的 `should_stop_` 标志
- 代码更简洁、可读性更强

### 4. **扩展性强**
- 可以轻松添加更多等待条件（如优先级变化、VM 事件等）
- 支持 `notify_all()` 批量唤醒
- 可与 `wait_until()` 配合实现精确定时

## 经验教训

### ✅ 最佳实践
1. **避免自旋轮询**：能用 condition_variable 就不要用 atomic+sleep
2. **单一职责**：停止逻辑集中在 `stop()` 方法中
3. **RAII 锁管理**：使用 `unique_lock` 而非手动 lock/unlock
4. **及时通知**：状态改变后立即 `notify_one()`

### ⚠️ 常见陷阱
1. **虚假唤醒**：condition_variable 可能无故唤醒，需要在循环中检查条件
2. **锁竞争**：等待时必须持有锁，但执行任务时应释放锁
3. **通知丢失**：先改变状态，再 notify，顺序不能颠倒
